## Notes: Polymorphic Type Identification without `<typeinfo>`

### 1. Making a Class Polymorphic for `dynamic_cast`
- `dynamic_cast` relies on **runtime type information (RTTI)**, which is stored in the **vtable**.
- A class only gets a vtable if it has at least **one virtual function**.
- Declaring `virtual ~Base() {}` is the minimal way to make `Base` polymorphic, enabling safe downcasting.
- A **pure virtual destructor** (`virtual ~Base() = 0;`) would make `Base` abstract and require an out‑of‑line definition, adding unnecessary complexity here. An ordinary virtual destructor is sufficient and simpler.

### 2. Type Identification without `<typeinfo>`
- `<typeinfo>` is forbidden, so `typeid` cannot be used.
- **Pointer version** (`identify(Base* p)`):
  Chain `dynamic_cast<Derived*>(p)` attempts. If the result is not `nullptr`, the actual type is known.
- **Reference version** (`identify(Base& p)`):
  Use `dynamic_cast<Derived&>(p)`. On failure it throws `std::bad_cast`.
  Since `std::bad_cast` is declared in `<typeinfo>`, we cannot catch it by name. Instead, we use `catch (...)` to catch any exception, then proceed to the next attempt.
  If neither `A&` nor `B&` succeeds, the object must be `C`.

### 3. Seeding the Random Generator Correctly
- `std::rand()` produces the same sequence every run unless seeded with `std::srand()`.
- The common mistake: calling `std::srand(std::time(0))` **every time** a random value is needed.
  If calls happen within the same second, the seed is identical, so `rand()` returns the same first number each time → all generated objects are of the same type.
- The fix: **seed only once**.
  A `static bool seeded` inside the generation function ensures that `srand` is called only on the first invocation, and never again. This keeps the random sequence advancing properly across multiple calls, producing varied results.
- Alternative (bureaucrat‑style): seed once in `main()` and rely on the caller’s discipline. The `static` guard is preferred for a self‑contained, reusable function.

### 4. Stack‑allocated vs. Heap‑allocated Pointer Arrays
- **`Base* arr[5];`**
  - The array itself lives on the **stack**; it is automatically destroyed when the scope ends.
  - Only the **pointed‑to objects** (allocated with `new`) need manual `delete`.
  - Must **not** use `delete[]` on the array.

- **`Base** arr = new Base*[5];`**
  - The array is allocated on the **heap** with `new[]`.
  - The `arr` variable itself (a pointer) is on the stack, but the memory it points to is on the heap.
  - Requires manual release of **both** the individual objects and the array:
    1. `delete` each `arr[i]` (for the derived objects).
    2. `delete[] arr` (for the pointer array).
  - Failure to do so causes memory leaks.

- **Key rule:** Every `new` must be balanced by a `delete`, every `new[]` by `delete[]`. Stack objects are managed automatically; never call `delete` on something not allocated with `new`.

### 5. `dynamic_cast` vs. `static_cast` – Why `static_cast` Cannot Replace `dynamic_cast` Here
- **`static_cast`** performs **compile‑time** conversions based on the static types known to the compiler.
  - It can downcast `Base*` to `A*`, but it does **no runtime check**.
  - If the object is not actually an `A`, the resulting pointer silently points to an invalid, incorrectly typed object. Dereferencing it leads to **undefined behavior**.
  - There is no “failure” indication (no `nullptr`, no exception) – the cast always “succeeds” syntactically.
- **`dynamic_cast`** performs **runtime** conversion that queries the object’s actual type via the vtable.
  - For pointers: returns `nullptr` if the object is not of the requested type.
  - For references: throws `std::bad_cast` on mismatch.
  - This allows **testing** the real type safely.
- **In this exercise**:
  - We need to **determine which derived type** the object truly is.
  - `static_cast` cannot give us that information because it never fails, even for a wrong type.
  - Only `dynamic_cast` provides the necessary success/failure feedback to identify the actual type. Therefore, `static_cast` cannot do the same thing.

### 6. Exception Flow in `identify(Base&)`
- When the first `dynamic_cast` throws, the corresponding `catch(...)` catches it and execution **continues normally** to the next `try` block.
- Only a successful cast triggers `return`, ending the function.
- This sequential try‑catch pattern effectively tests each possible type without relying on `typeid`.
