## 📘 ScalarConverter Core Notes (Concept + Syntax + Test Edge Cases)

### I. Core Design Philosophy (Architecture)
**Single Source of Truth (Unified Intermediate)**
- All input strings are first parsed into a `double`, then converted to `char`, `int`, and `float` for display.
- **Why `double`?** It has the largest range and highest precision among the target types. It can safely hold `int`, `float`, and special values (`nan`/`inf`). This avoids parsing the same string 4 separate times.

**Separation of Concerns**
- `isInt`, `isFloat`, etc.: Responsible **only for format validation** (does this string *look like* an X?).
- `convert` (main function): Responsible for **actual parsing** and storing the result as a `double`.
- `convertToChar`, `convertToInt`, etc.: Responsible for **formatting and printing** the output.

---

### II. Key Syntax & Hidden Pitfalls

#### 1. How `std::stringstream` Works
- Skips leading whitespace (spaces, `\t`).
- Reads until it hits a non-numeric character (e.g., `"123abc"` reads `123` and leaves `"abc"` in the stream).
- **State Checks**:
  - `ss.fail()`: Extraction failed (e.g., parsing `"abc"`).
  - `ss.eof()`: Whether the pointer has reached the end of the string (used to reject `"123abc"`).

#### 2. The Safe `isInt` Implementation (Must Memorize)
```cpp
std::stringstream ss(str);
long long num;          // Use long long first to avoid Undefined Behavior (UB) on int overflow
ss >> num;
if (ss.fail() || !ss.eof()) return false;
if (num < INT_MIN || num > INT_MAX) return false;  // Manual range check
return true;
```
- **Why not `int` directly?** Directly using `ss >> int` when the value overflows results in **Undefined Behavior (UB)**. Using `long long` and manually checking the range against `INT_MIN`/`INT_MAX` is the standard, safe practice.

#### 3. Handling Pseudo-Literals (`nan`, `inf`)
- You **must** manually check for `"nan"`, `"nanf"`, `"inf"`, `"inff"`, `"+inf"`, `"-inff"`, etc.
- **Why?** When parsing `"nanf"` with `stringstream`, some standard libraries extract `nan` but leave the `'f'` behind, causing `!eof()` and a false rejection. Manual checks ensure cross-platform consistency.
- **Assignment of special values**:
  ```cpp
  std::numeric_limits<double>::quiet_NaN();   // Quiet NaN
  std::numeric_limits<double>::infinity();    // Positive infinity
  -std::numeric_limits<double>::infinity();   // Negative infinity
  ```

#### 4. Floating-Point Output Precision Control
- **Required Header**: `#include <iomanip>`
- **`std::fixed`**: Forces fixed-point notation (no scientific notation like `1e+02`).
- **`std::setprecision(n)`**:
  - Combined with `std::fixed` -> `n` digits after the decimal point.
  - Without `std::fixed` -> `n` significant digits total (`float` usually uses 7, `double` uses 15-16).

#### 5. Library Functions for Special Values
- **Required Header**: `#include <cmath>`
- `std::isnan(value)`: Checks if the value is NaN.
- `std::isinf(value)`: Checks if the value is infinite (positive or negative).

---

### III. Why Check Before `static_cast`? (Crucial)

| Scenario | Consequence of Direct `static_cast` |
| :--- | :--- |
| `value` is `NaN` or `inf` | **Undefined Behavior (UB)** - program may crash. |
| `value` exceeds `char`/`int` range | **Undefined Behavior (UB)** - gets truncated or yields a random value. |
| `value` has a fractional part (e.g., 42.9) | **Legally truncates to 42**, but business logic requires `"impossible"`. |

**Rule**: Always check first. Order of checks: `isnan/isinf` -> Range check -> Fractional part check (if required) -> Displayability (for `char`).

### IV. Critical Test Cases (Must Check Before Submission)

#### 1. Boundary Values (Edge Limits)
- **`int` boundaries**: `"2147483647"` (should succeed), `"2147483648"` (should be rejected by `isInt` because > INT_MAX, but **must NOT be accepted by `isDouble` either** since it has no `.` or `e` -> fall to `else`).
- **`long long` boundaries**: `"9223372036854775807"` (should be rejected by `isInt`, and also rejected by `isDouble` as pure integer -> fall to `else`).
- **Exceeds `long long`**: `"999999999999999999999"` (parsing into `long long` will `fail`. Should fall into the final `else` and print `"impossible"`).

#### 2. Suffixes and Decimal Points
- `"42.0f"` (valid `float`).
- `"42f"` (**Strict C++**: INVALID, reject; **GCC extension**: VALID, accept. Clarify your project rule).
- `"1e3f"` (**ALWAYS VALID**: has exponent `e` + suffix `f`).
- `"42.0"` (valid `double`).
- `"42."` / `".5"` (valid C++ `double`, with decimal point).
- `"1e3"` (valid `double`, scientific notation).

#### 3. Whitespace Characters
- `"  42"` (leading spaces): If your `isInt` relies on `ss.eof()`, it will **incorrectly accept it** because `>>` skips leading spaces.
- **To reject leading/trailing spaces**, you must manually check the first character (or all characters) or use `std::strtod` which allows ignoring trailing spaces. Clarify your project's requirement on whitespace.

#### 4. Scientific Notation (Very Common Trap)
- `"1e3"` (1000), `"1.5e-2"` (0.015).
- If your `isDouble` only checks for `.` or `f`, it will **miss `"1e3"`**. You must check for `'e'`/`'E'` as well.

#### 5. Character Input
- Format: `'c'` (single quotes).
- **Non-displayable characters**: ASCII 0–31 and 127. Must output `"Non displayable"`.
- **Displayable characters**: ASCII 32 (space) to 126 (`~`). Must output `'A'`, `' '`, etc.

#### 6. Pseudo-Literal Capitalization
- C standards support `"NAN"` and `"INF"`, but C++ streams usually don't. Decide on a rule (e.g., only lowercase) and stick to it, or handle both.

---

### V. The Final Defense (The Catch-All `else`)
```cpp
else {
    std::cout << "char: impossible" << std::endl;
    // ... print for int, float, double
}
```
- **Must be present!** It handles completely invalid input like `"abc"`, `"+"`, `""`, etc., that none of the `is*` functions recognize.

---

### VI. Common Interview/Exam Questions
- **Q: Why not use `atoi` / `atof`?**
  **A:** They cannot distinguish between `"0"` and a parsing failure (e.g., `"abc"` returns 0). Also, they provide no overflow checking.

- **Q: What is the memory difference between `float` and `double`?**
  **A:** `float` is 32 bits (~7 decimal digits of precision), `double` is 64 bits (~15-16 decimal digits). Casting `double` to `float` loses precision, which is handled by `static_cast`.

### VII. Float / Double Literal Syntax (Standard C++) ⭐ NEW
This is the **most common source of logic bugs**. You must implement `isFloat` and `isDouble` based on **syntactic form**, not just "can stringstream parse it".

#### A. C++ Standard Rules (ISO C++)

| Feature | `double` Literal | `float` Literal |
| :--- | :--- | :--- |
| **Must have** | `.` (decimal point) **OR** `e`/`E` (exponent). | `.` **OR** `e`/`E` (in the core part) **AND** suffix `f`/`F`. |
| **Cannot have** | Suffix `f`/`F`. | — |
| **Pure integer** (`42`, `2147483648`) | ❌ **NOT a double**. Even if it exceeds `int`, it is still an integer literal (type `long long`). | ❌ **NOT a float**. |

#### B. Critical Edge Cases (Must Code Correctly)

| Input | Standard C++ Type | Your `isDouble` | Your `isFloat` | Note |
| :--- | :--- | :--- | :--- | :--- |
| `"42"` | Integer | `false` | `false` | No `.` or `e`. |
| `"2147483648"` | Integer (`long long`) | `false` | `false` | No `.` or `e`. Even if > INT_MAX, do NOT treat as double. |
| `"42.0"` | `double` | `true` | `false` | Has `.`, no `f`. |
| `"1e3"` | `double` | `true` | `false` | Has `e`, no `f`. |
| `"42.0f"` | `float` | `false` (reject `f`) | `true` | Has `.` + `f`. |
| `"1e3f"` | `float` | `false` (reject `f`) | `true` | Has `e` + `f`. ✅ **Must support!** |
| `"42f"` | **Invalid** (GCC extension) | `false` | **Depends on project**: Strict = `false`, Extended = `true` | Standard says NO because no `.` or `e`. |


#### C. Summary of the Golden Rule
> **Pure digits (no `.`, no `e`, no `f`) = Integer forever.**
> **If it has `.` or `e` but no `f` = `double`.**
> **If it has `f` AND (`.` or `e`) = `float`.**
> **If it has `f` but no `.` and no `e` = Invalid (unless you choose to support GCC extension).**
